# Species richness mapping (hex grid, skates, rays, sharks)

Self-contained pipeline for global batoid species-richness maps and latitude diagnostics. Open the **repository root** in RStudio (or Cursor) so `here::here()` resolves paths; do not set the working directory only to `mapping/`.

## Components

| Piece | Role |
|--------|------|
| **`00_functions.R`** | `calculate_hex_metrics2()`, `plot_species_richness_map()`, latitude label helpers for `coord_sf`. |
| **`HexCells/`** | Cylindrical equal-area hex shapefile; used if `data/processed/hex.rds` is missing. |
| **`01_SpatialDataProcessing.Rmd`** | Builds `data/processed/hex.rds`, `ranges.rds`, optional `hex_richness.rds`, and **`hex_with_metrics_skates.rds` / `hex_with_metrics_rays.rds`**. Requires **`chondrichthyes.species.ranges`** and parallel packages; can be slow. |
| **`02_SpeciesRichnessMaps.Rmd`** | Stacked skate/ray richness maps; writes PNG/PDF under **`mapping/figures/`**. |
| **`03_SkateRayRichnessPublication.Rmd`** | Four-panel publication figure (maps + latitude scatters); high-resolution PNG to **`mapping/figures/`**. |
| **`03b_SkateRayRichnessPublicationReversed.Rmd`** | Same as `03_` with reversed viridis on fill and point colour. |
| **`04_RichnessLatitude.Rmd`** / **`04b_…Reversed.Rmd`** | Richness–latitude analyses and extra figure exports. |
| **`06_SharkRichness.Rmd`** | Shark richness maps and hexbin figures. |

## Suggested run order

1. Install R packages (see below).
2. **`01_SpatialDataProcessing.Rmd`** — creates **`data/processed/`** caches (or copy `.rds` files from another machine that already ran this step).
3. **`02_` … `06_`** as needed.

## Data layout (repository root)

- **`data/processed/*.rds`** — generated by `01_` (large; listed in `.gitignore` in this repo). See **`data/processed/README.md`**.
- **`mapping/figures/`** — mapping outputs (PNGs/PDFs from `02_`–`06_`).

## R packages

Core: `sf`, `dplyr`, `tidyr`, `ggplot2`, `here`, `patchwork`, `rnaturalearth`, `purrr`, `future`, `furrr`, `future.apply`.

Spatial ranges: **`chondrichthyes.species.ranges`** (e.g. `remotes::install_github("Marine-Biodiversity-Conservation-Lab/chondrichthyes.species.ranges")`).

Optional / script-specific: `tidyverse`, `hexbin` (shark hexbin chunk in `06_`).

## Parallel processing (`future` / `furrr`)

Heavy steps (`01_SpatialDataProcessing.Rmd`, `calculate_hex_metrics2()` in **`00_functions.R`**) use **future**-style parallelism. You set a **plan** for the R session; it applies to `furrr::future_*` and `future.apply` calls.

### All platforms: multisession (recommended default)

**Multisession** starts separate R worker processes. It works on **macOS, Linux, and Windows** and matches the approach in `01_SpatialDataProcessing.Rmd` (hex–range intersections):

```r
library(future)
plan(multisession, workers = max(1, parallel::detectCores() - 1))
```

When parallel work is done, return to a single process (as `01_` does) with:

```r
plan(sequential)
```

**Choosing `workers`:** `parallel::detectCores() - 1` leaves one core free for the OS. If RAM fills up (each worker holds a full R process), reduce workers (e.g. `workers = 4`) or close other applications.

### macOS and Linux only: optional `multicore` (forking)

On **Unix-like** systems (macOS, Linux), you can try **`plan(multicore, ...)`**, which forks the current R process. Forking can be faster or more memory-efficient for some tasks, but:

- It is **not available on Windows**; Windows users should keep **`multisession`**.
- It can interact badly with some GUIs, packages, or BLAS threads; if you see hangs or crashes, switch back to **`multisession`**.

Example that picks a strategy by OS:

```r
library(future)
w <- max(1, parallel::detectCores() - 1)
if (.Platform$OS.type == "windows") {
  plan(multisession, workers = w)
} else {
  plan(multicore, workers = w)   # try multisession here if multicore misbehaves
}
```

### RStudio / troubleshooting

If workers fail to start or jobs stall in **RStudio**, run the document from a plain R session or the shell (still from the **repository root**):

```r
rmarkdown::render("mapping/01_SpatialDataProcessing.Rmd")
```

### Clusters (optional)

On an HPC cluster you can use **`future::plan(cluster, workers = ...)`** with **`parallel::makeCluster()`**; that is optional for typical laptop runs.
